# Ours 算法结构并行性利用分析

> 日期：2026-03-17
> 背景：v13（Conv/FC 分类型张量并行）后，分析当前算法的加速来源和结构并行性利用瓶颈

---

## 一、核心发现：加速几乎完全来自张量并行，而非结构并行

### 1.1 服务器利用率数据

| 模型 | 延迟 (ms) | Server 0 利用率 | Server 1-3 利用率 | 含义 |
|------|----------|----------------|------------------|------|
| InceptionV3 | 890.7 | **99.4%** (885ms) | 21-24% (~200ms) | 主链串行在 S0，仅 TP shards 用其他服务器 |
| YOLOv5 | 8304.4 | **100%** (8304ms) | 15.4% (~1283ms) | backbone 完全串行在 S0 |
| VGG-16 | 1353.4 | ~100% | ~15% | 全串行链，TP shards 分布 |

**结论**：4 台服务器中，只有 Server 0 在满负荷工作，其余 3 台仅处理张量并行 shard——工作负载极度不均衡。

### 1.2 关键路径 vs 结构并行度

| 模型 | 总计算量 (ms) | 关键路径 (ms) | 并行度比值 | 理论最优 (4 srv) | 当前 Ours | 差距 |
|------|-------------|-------------|----------|----------------|----------|------|
| InceptionV3 | 1505.8 | 920.9 | **1.64×** | 376.4 | 890.7 | 890.7 < CP（TP 缩短了关键路径） |
| YOLOv5 | 11613.1 | 11412.2 | **1.02×** | 2903.3 | 8304.4 | 结构并行度几乎为零 |
| VGG-16 | 2261.7 | 2261.7 | **1.00×** | 565.4 | 1353.4 | 全串行，TP 是唯一加速来源 |
| BERT-large | ~2300 | ~2300 | **1.00×** | ~575 | 2287.1 | 全串行 |

**关键发现**：
- **绝大多数 DNN 模型的结构并行度极低**（≤1.02×），只有 InceptionV3 (1.64×) 有中等程度的 DAG 并行性
- **InceptionV3 的 Ours (890.7ms) 已低于无 TP 的关键路径 (920.9ms)**，说明 TP 缩短了关键路径本身
- YOLOv5 和 VGG-16 的加速 **100% 来自张量并行**（缩短单层计算时间），与结构并行无关

### 1.3 张量并行 vs 结构并行的加速分解

| 模型 | MEDIA (ms) | Ours (ms) | Ours 加速来源 |
|------|-----------|----------|--------------|
| InceptionV3 | 1403.1 | 890.7 | TP (缩短 30 个 Conv 层) + HEFT 调度（分支分配到不同服务器） |
| YOLOv5 | 11432.8 | 8304.4 | **纯 TP**（42 个 Conv 层加速），结构并行贡献约 0% |
| VGG-16 | 2957.1 | 1353.4 | **纯 TP**（12 个 Conv/FC 层加速），结构并行贡献 0% |
| BERT-large | 2306.7 | 2287.1 | 无 TP 候选（层太小），保底触发 |

## 二、HEFT 调度器对结构并行的利用现状

### 2.1 HEFT 的工作方式

当前 Ours 使用标准 HEFT（Heterogeneous Earliest Finish Time）调度：

1. 按优先级排序所有分区（基于 bottom-level 权重）
2. 逐个分区贪心选择「最早完成时间」的服务器
3. 考虑依赖约束（等待所有前驱完成 + 通信延迟）

### 2.2 HEFT 做对了什么

从 InceptionV3 的调度结果可以看到，**TP shards 确实被分配到了不同服务器**：

```
reduction_a_b2_3x3_stride2_shard_1 → Server 3 [681.9-687.7ms]
reduction_a_b2_3x3_stride2_shard_2 → Server 2 [686.7-692.5ms]
reduction_a_b2_3x3_stride2_shard_3 → Server 3 [687.7-693.5ms]
...
```

HEFT 的贪心策略自然将同一层的 shards 分散到不同服务器——因为 Server 0 忙于串行链，其他服务器空闲，「最早完成时间」自然指向它们。

### 2.3 HEFT 做不到什么

**问题一：串行主链独占 Server 0**

InceptionV3 的 stem（卷积链）约 600ms，完全串行，只能在一台服务器上执行。HEFT 将其分配到 Server 0 后，Server 0 在 0-600ms 期间被独占。其他服务器空闲等待。

**问题二：InceptionV3 的 DAG 分支发生在深层**

Inception 块的 fork/join 结构位于 stem 之后（600ms+），此时只剩 ~300ms 的工作量。即使完美并行化这 300ms 中并行度为 1.64× 的部分，收益也只有 300 × (1 - 1/1.64) ≈ 117ms——而 TP 已经通过缩短层级计算实现了更大的收益。

**问题三：分区图存在环路**

当前 MEDIA-style 分区 + 合并后的分区图包含 230 个环路。这是因为层合并可能将双向依赖的层组合成不同分区，导致分区间产生循环依赖。虽然 HEFT 调度器通过优先级排序规避了死锁，但这意味着分区图并非严格 DAG，限制了进一步的拓扑分析和优化。

## 三、优化可能性分析

### 3.1 方向一：关键路径感知的 TP 并行度选择

**当前问题**：DP 为每个层独立选择最优 k，不区分该层是否在关键路径上。

**优化思路**：优先将 TP 资源（多服务器 shard）分配给关键路径上的算子，因为缩短关键路径直接降低总延迟，而缩短非关键路径上的算子可能无效。

**预期收益**：中等。InceptionV3 的关键路径已被 TP 覆盖（890ms < 920ms CP），但可以进一步优化 k 值分配。

**可行性**：★★★★☆
实现简单——在 DP 阶段增加关键路径权重因子。但需要两轮 DP：第一轮找 CP，第二轮加权优化。

### 3.2 方向二：分支感知 HEFT 调度

**当前问题**：HEFT 逐个调度分区，不主动将 fork 点的后继分配到不同服务器。

**优化思路**：在 HEFT 中引入「分支感知」启发式——当检测到分区有多个无依赖的后继（fork 点）时，主动将后继分散到不同服务器，即使某些服务器的「最早完成时间」不是最优。

**预期收益**：
- InceptionV3：中等（DAG 并行度 1.64×，但分支工作量仅占 ~300ms / 890ms）
- YOLOv5：极低（DAG 并行度仅 1.02×，几乎无分支可并行）
- BERT/VGG-16：零（全串行）

**可行性**：★★★☆☆
需要修改 HEFT 逻辑，但需解决分区图有环的问题。且实际收益取决于模型结构——对当前实验模型集，仅 InceptionV3 有可观收益。

### 3.3 方向三：分区级流水并行（Partition Pipeline）

**当前问题**：每个分区必须等前驱分区完全执行完毕才能开始。

**优化思路**：对于串行链中的相邻分区，引入流水线重叠——当前分区的最后几层还在计算时，后继分区的前几层已开始在另一台服务器上执行（只要数据依赖已满足）。

**预期收益**：
- 对串行长链模型（YOLOv5, VGG-16, BERT）有潜在收益
- 但需要层级（而非分区级）的细粒度调度，调度开销增大
- 通信延迟可能抵消流水线收益（特别是在 100Mbps 带宽下）

**可行性**：★★☆☆☆
需要重构调度器为层级调度，复杂度显著增加。且 SGX enclave 的上下文切换开销可能使细粒度流水线不实际。

### 3.4 方向四：混合策略——TP + 数据并行

**当前问题**：当前只考虑单次推理。实际边缘场景可能有多个推理请求。

**优化思路**：对于串行模型（BERT, VGG-16），不做单次推理的并行化，而是在多请求场景下做数据并行——不同服务器处理不同推理请求。

**预期收益**：吞吐量线性提升（4×），但单次延迟不变。

**可行性**：★★★★★
实现简单，但改变了优化目标（从延迟到吞吐量），与论文当前 focus 不同。可在 Discussion 中提及。

### 3.5 方向五：算子融合 + 更大分区粒度的 TP

**当前问题**：TP 以单层为粒度，小算子（如 BN, SiLU）无法受益。

**优化思路**：将相邻算子融合为复合算子（如 Conv-BN-SiLU），然后对复合算子做 TP。这增大了单次 TP 的计算量，使 TP 的 overhead（AllGather/AllReduce）相对更低。

**预期收益**：
- YOLOv5：每个 Conv-BN-SiLU 三元组融合后，TP 候选增加
- 对已有候选，收益有限（当前已选择大算子）

**可行性**：★★★☆☆
需要在 augment graph 阶段实现算子融合逻辑。但融合规则需要考虑不同算子类型的张量布局兼容性。

## 四、结论与建议

### 4.1 核心结论

**当前 Ours 算法以张量并行为主要加速手段，这在策略上是正确的**。原因：

1. **模型结构并行度天然有限**：7 个实验模型中只有 InceptionV3 (1.64×) 有中等结构并行度，其余模型均 ≤ 1.02×
2. **TP 能加速串行模型**：VGG-16 和 YOLOv5 完全串行，只有 TP 能缩短延迟
3. **TP 缩短关键路径本身**：即使模型有并行结构，关键路径上的大算子仍然是瓶颈，TP 直接缩短它们
4. **InceptionV3 上 Ours 已逼近关键路径下界**：890.7ms vs CP=920.9ms（甚至低于无 TP 的 CP）

### 4.2 对论文的启示

1. **在 Design 章节中明确定位**：Ours 的核心创新是类型感知张量并行（Conv AllGather / FC AllReduce），而非 DAG 并行利用。HEFT 调度是标准方法，负责将 shards 分配到服务器。
2. **在 Evaluation 中增加分析**：展示关键路径 vs Ours 延迟的对比，说明 TP 缩短了 CP 本身。
3. **在 Discussion 中讨论局限性**：
   - 当前方法对结构并行的利用有限，但这是因为主流 DNN 模型本身结构并行度低
   - InceptionV3 是特例（1.64×），但即使在该模型上，TP 的贡献也大于结构并行
   - 未来工作可以考虑分支感知调度和关键路径优先 TP，特别是对于具有更高结构并行度的新兴模型（如 Mixture of Experts）

### 4.3 优化优先级排序

| 优先级 | 方向 | 预期收益 | 实现复杂度 | 建议 |
|--------|------|---------|-----------|------|
| 1 | 关键路径感知 TP | 中 | 低 | **值得实现**——在 DP 阶段加入 CP 权重 |
| 2 | 算子融合 | 中 | 中 | 可在论文中描述为 future work |
| 3 | 分支感知 HEFT | 低（仅 InceptionV3） | 中 | 论文 Discussion 提及即可 |
| 4 | 分区级流水线 | 低 | 高 | 不建议当前实现 |
| 5 | 数据并行（多请求） | 高（吞吐量） | 低 | Discussion 中提及 |
